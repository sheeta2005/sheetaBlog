# Redis持久化
## 一、概述
Redis是内存数据库，断电/进程宕机内存数据全部丢失，**持久化就是把内存数据落地磁盘，故障重启后从磁盘文件恢复数据**，提供两种方案：**RDB快照、AOF日志**，生产环境推荐双开。

---
## 二、RDB（Redis Database，数据快照）
### 1.定义
**全量快照**：某一时刻把Redis全量内存数据压缩写入`.rdb`二进制文件，恢复时直接加载整份快照。
### 2.三种触发方式
1. **save**：主进程同步执行，全程阻塞Redis所有读写命令，生产禁用；
2. **bgsave**：后台异步，主进程`fork`子进程做落地，不阻塞主线程（主流）；
3. **配置自动触发（redis.conf）**
```conf
save 900 1    # 900s内≥1次key修改，自动bgsave
save 300 10   # 300s内≥10次key修改，自动bgsave
save 60 10000 # 60s内≥10000次key修改，自动bgsave
```
### 3.fork+写时复制（COW）底层原理
1. `bgsave`触发时，操作系统fork生成子进程，**子进程和主进程共用同一份物理内存页表**；
2. 子进程遍历内存、写入RDB磁盘文件；
3. **主进程读数据**：继续共享原有内存；
4. **主进程写数据**：触发COW，复制被修改的内存页，在新副本上修改，子进程仍保留原始快照数据，保证快照数据一致性。

### 4.RDB优缺点
✅优点：文件压缩体积小、宕机数据恢复速度极快；
⚠️缺点：两次快照中间新增的数据无落地，故障会丢失区间全部数据。

---
## 三、AOF（Append Only File，追加命令日志）
### 1.定义
记录**每一条变更类写命令（set/del/hset等）**，以Redis协议格式追加存入`.aof`文本文件，重启时逐条重放命令还原数据，默认关闭。
开启配置：
```conf
appendonly yes
appendfilename "appendonly.aof"
```
### 2.三种刷盘策略（appendfsync）
|配置|刷盘时机|优点|缺点|
|----|----|----|----|
|always|写完命令立刻落盘|数据最安全，几乎不丢数据|频繁磁盘IO，性能极差|
|everysec（默认）|缓冲区每秒统一刷盘|性能均衡|最多丢失1秒数据（生产标配）|
|no|交给OS自主调度刷盘|性能最优|极端宕机丢失大量数据|

### 3.AOF重写（AOF rewrite）
海量重复命令会让AOF文件膨胀，重写会**根据当前内存数据生成最简指令**，生成新AOF文件替换旧文件，压缩体积。

### 4.AOF优缺点
✅优点：数据完整性高，按秒落地，丢数据极少；
⚠️缺点：命令日志体积远大于RDB，海量指令重放时恢复速度很慢。

---
## 四、RDB vs AOF 对比汇总
|对比项|RDB|AOF|
|----|----|----|
|持久化形式|定时全量内存快照|实时追加写命令日志|
|数据完整度|差，两次快照间数据全丢|高，最多丢1s(默认配置)|
|文件体积|二进制压缩，体积小|文本存指令，体积大|
|恢复速度|极快，直接加载全量数据|很慢，逐条回放命令|
|资源开销|fork子进程，CPU开销偏高|磁盘IO频繁，IO开销偏高|
|适用场景|可容忍分钟级丢数据、追求快速启动|高数据安全、不能大量丢数据|

## 五、生产落地规范
**RDB+AOF同时开启**：
1. 优先用AOF做故障数据恢复（数据最全）；
2. RDB用作冷备份、跨机器数据迁移，兼顾恢复速度与数据安全。

---
## 六、面试延伸（对应开篇面试提问）
面试官问：项目Redis缓存怎么保障数据不丢失？
答：生产双开RDB+AOF持久化，everysec刷盘；RDB定时自动快照，AOF保障秒级数据落地，宕机优先AOF恢复，RDB做异地备份。

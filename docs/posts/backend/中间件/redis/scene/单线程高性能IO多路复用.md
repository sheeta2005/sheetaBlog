# Redis单线程高性能 + IO多路复用笔记
存放路径：`redis/scene/Redis高性能原理&IO多路复用.md`
## 一、面试题：Redis单线程为什么速度极快
1. **纯内存操作**
   所有数据常驻内存，避免磁盘IO耗时，内存读写纳秒级速度，瓶颈在网络而非数据运算。
2. **单线程架构**
   规避多线程上下文切换、锁竞争、线程安全开销，CPU资源全部用于数据处理。
3. **IO多路复用（epoll）非阻塞网络模型**
   单线程同时监听成千上百客户端Socket，空闲不空转，高效处理海量并发连接。

## 二、前置：Linux用户空间&内核空间
1. **用户空间(Ring3)**：应用程序运行区域，无硬件权限，不能直接操作硬件。
2. **内核空间(Ring0)**：操作系统内核，拥有硬件操作特权，管理网卡、磁盘。
3. IO数据流转：
   读：硬件→内核缓冲区→用户缓冲区；写：用户缓冲区→内核缓冲区→硬件。所有IO必须经过内核中转。

## 三、Linux四种IO模型对比
### 1. 阻塞IO Blocking IO
- 两个阶段全程阻塞：
  ①等待数据到达内核缓冲区：进程卡死等待网卡数据；
  ②内核拷贝数据到用户空间：进程继续阻塞。
- 缺点：一个线程只能处理一个连接，并发极低。

### 2. 非阻塞IO Non-Blocking IO
- 阶段一：无数据立刻返回错误码，进程循环轮询`recvfrom`，CPU空转浪费资源；
- 阶段二：数据就绪后，内核拷贝数据时进程依旧阻塞；
- 缺点：频繁系统调用，CPU占用飙升，不适合高并发。

### 3. IO多路复用（Redis采用epoll实现）
**定义：单个线程监听批量Socket，任意Socket就绪就通知进程，避免无效轮询等待**
1. **执行阶段**
   阶段1：进程调用`select/poll/epoll`，阻塞等待内核监听的一批socket，任一就绪返回可读；
   阶段2：遍历就绪socket，调用`recvfrom`拷贝内核数据至用户空间。
2. 三种多路复用实现区别
   | 实现 | 特点 |
   | ---- | ---- |
   | select/poll | 只通知有连接就绪，不指明具体fd，用户需要全量遍历所有socket |
   | epoll（Redis默认） | 就绪时直接把就绪socket写入用户空间，无需遍历，海量连接性能最优 |

## 四、Redis网络事件模型
### 1. 三大事件处理器
1. **连接应答处理器（tcpAcceptHandler）**：监听端口，接收客户端TCP连接。
2. **命令请求处理器（readQueryFromClient）**：读取客户端命令，6.0版本多线程做命令解析，**命令执行仍单线程串行**。
3. **命令回复处理器（sendReplyToClient）**：把结果写回客户端，Redis6+多线程异步发送返回数据。

### 2. 整体流程
客户端socket → epoll监听 → 事件分发 → 三类处理器分工处理 → 执行结果入缓冲区 → 多线程回包。

## 五、总结面试简答
1. Redis快三点：全内存、单线程无切换开销、epoll多路复用高并发收发包。
2. IO多路复用：单线程监听海量socket，就绪通知，epoll精准返回就绪fd，高效利用CPU。

# 双写一致性全方案汇总
## 一、基础定义
双写一致性：**MySQL数据发生增删改时，同步维护Redis缓存，保证DB与缓存数据相同，避免缓存遗留旧脏数据**。
### 读操作通用逻辑
请求优先查询Redis：
1. 缓存命中 → 直接返回数据；
2. 缓存未命中 → 查询MySQL，结果回写入Redis并配置过期时间后返回。

## 二、方案1：延时双删（同步方案，中小项目常用）
### 执行顺序：第一次删缓存 → 更新MySQL → 延时等待 → 第二次删缓存
1. **首次删缓存**：清空旧缓存，防止后续读请求命中过期数据；
2. **更新数据库**：落地最新数据至MySQL；
3. **延时二次删缓存**：等待「并发读请求查库+回填缓存」、主从数据同步完成，删除并发写入的脏缓存。
> ⚠️缺点：仍存在极小概率脏数据，仅能做到**最终一致**，无法强一致。

## 三、方案2：读写锁方案（读多写少业务，强一致性首选）
依托Redisson**读写分布式锁**，规则：**读读共享、读写互斥、写写互斥**
1. **写流程（加排他WriteLock）**
   加排他锁 → 修改数据库 → 删除缓存 → 释放锁；
   加锁期间，所有读请求阻塞，杜绝改库间隙旧数据回填缓存。
2. **读流程（加共享ReadLock）**
   加共享锁 → 查询缓存，缓存失效则查库并更新缓存 → 释放锁；
   多个读线程可同时加读锁并发读取，不会互相阻塞。
> ✅优点：强数据一致；缺点：写操作阻塞大量读，高并发写场景性能差，适配商品详情、资讯等读多写少业务。

## 四、方案3：MQ异步更新缓存（分布式中大型项目，异步最终一致）
### 流程
1. item-service修改数据，**先落库MySQL，再发送缓存更新消息到MQ**；
2. 独立cache-service消费MQ消息，执行删除/更新Redis缓存；
> 关键点：必须保障MQ消息可靠性（生产者确认+消息持久化+死信队列），防止消息丢失导致缓存不更新。
- 优点：业务线程不阻塞，解耦读写服务，失败可重试；
- 缺点：依赖MQ中间件，架构变复杂。

## 五、方案4：Canal监听Binlog（零侵入业务，大厂主流）
### 流程
1. item-service正常修改数据写入MySQL，**业务代码完全不用操作缓存、不用发消息**；
2. Canal伪装成MySQL从节点，实时监听MySQL binlog二进制日志，捕获数据变更事件；
3. Canal推送变更数据 → cache-service收到通知 → 更新/删除Redis缓存。
> ✅核心优势：**完全解耦业务代码**，改数据不用关心缓存逻辑，业务故障不影响缓存更新；
> ⚠️缺点：引入Canal中间件，运维成本提升。

## 六、四种方案选型总结
|方案|适用场景|一致性|性能|
| ---- | ---- | ---- | ---- |
|延时双删|单体小型项目、低并发|最终一致|优秀|
|读写锁|读多写少、要求强一致（商品/资讯）|强一致|写阻塞读，写多场景性能差|
|MQ异步|中大型分布式项目、高并发|最终一致|优秀，异步解耦|
|Canal Binlog|大型分布式、零代码侵入需求|最终一致|最优，业务无感知|

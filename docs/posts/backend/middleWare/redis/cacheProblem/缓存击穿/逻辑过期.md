# 方案名称：**逻辑过期方案（解决缓存击穿）**
## 一、核心思路
1. **Redis不设置物理TTL（不过期删除key）**，在value里额外存一个`expire`逻辑过期时间戳。
2. 缓存数据永远留在Redis，不会被Redis主动清掉。

## 二、流程图分步拆解
1. **线程1（拿到分布式锁）**
   ① 查询缓存，对比当前时间＞expire → 判定逻辑过期
   ② 获取分布式互斥锁成功
   ③ **新开异步子线程去查询DB、更新Redis缓存、刷新expire过期时间**
   ④ **主线程直接返回旧的过期缓存数据**，不阻塞用户

2. **线程2、3（抢锁失败）**
   拿不到锁，直接返回当前旧缓存数据，**不会去访问数据库**。

3. **线程4（后续请求）**
   异步线程已经刷新完缓存，expire时间更新，直接命中有效缓存。

## 三、优缺点
### 优点
1. **全程不阻塞用户**，所有请求都能返回数据，吞吐量高
2. 杜绝大量并发同时打DB，完美防击穿
### 缺点
1. 会短暂返回过期数据，**无法保证实时一致性**（适合允许短时间脏读的业务：商品首页、热点文章）

## 四、和互斥锁方案对比
- **互斥锁**：拿不到锁阻塞等待，强一致性，用户等待耗时
- **逻辑过期**：拿不到锁直接返旧数据，无等待，容忍短暂数据不一致

## 补充
底层依赖Redisson分布式锁实现。

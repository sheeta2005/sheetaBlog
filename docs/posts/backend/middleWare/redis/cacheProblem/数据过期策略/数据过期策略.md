# Redis过期Key删除策略
## 一、核心结论
Redis**过期key不会立刻物理删除**，采用「**惰性删除 + 定期删除**」双策略配合管控过期数据。

## 二、惰性删除（被动删除）
### 原理
key过期后保留在内存，**只有客户端执行get/hmget等访问指令时**，Redis校验过期时间，过期则当场删除并返回空；未过期正常返回数据。
### 优点
CPU开销极低，闲置过期键不占用CPU扫描资源。
### 缺点
长期无人访问的过期key永久驻留内存，持续占用内存空间，容易出现内存泄漏。

## 三、定期删除（主动扫描）
### 原理
Redis按固定频次随机抽取部分key批量校验、清理过期数据，分SLOW、FAST两套执行模式：
1. **SLOW模式**：定时任务，默认`hz=10`（10次/秒），单次扫描耗时上限25ms，可在`redis.conf`修改hz参数调整频率；
2. **FAST模式**：事件驱动触发，间隔≥2ms，单次执行耗时≤1ms，CPU占用更克制。
> 逻辑：随机取样而非全库遍历，平衡内存回收与CPU损耗。
### 优点
主动释放无人访问的过期key，弥补惰性删除内存浪费问题；可限制单次执行时长，避免CPU打满。
### 缺点
取样随机性，仍有小概率部分过期key长期没被扫到、残留内存；参数调优难度高。

## 四、二者组合逻辑
1. **惰性兜底**：热点过期key被访问时立刻删除；
2. **定期兜底**：冷门长期闲置的过期key靠定时巡检逐步清理；
3. 补充兜底：内存达到上限时触发**内存淘汰策略**，强制清理数据（淘汰是独立机制，和过期删除无关）。

## 五、面试标准回答话术
Redis过期key不会立即删除，采用惰性+定期组合：
1. 惰性删除：访问key时校验过期，过期即删，节省CPU但会堆积冷过期数据；
2. 定期删除：分SLOW/FAST定时随机抽样清理过期键，释放内存、弥补惰性短板；
   二者搭配兼顾CPU与内存平衡。

# 循环引用

## 一、核心基础概念补充
### 1. 什么是循环依赖？
循环依赖即循环引用，指两个或以上Bean互相持有对方，形成闭环依赖（如A依赖B，B依赖A）。

### 2. 什么是单例池？
单例池即Spring的**一级缓存（singletonObjects）**，是`DefaultSingletonBeanRegistry`中的`ConcurrentHashMap`，用于缓存已完成完整生命周期、初始化完成的单例Bean对象。

### 3. 为什么三级缓存优于二级和一级？
- 一级缓存（singletonObjects）仅存储完全初始化的Bean，无法解决循环依赖中“半成品对象”的注入问题；
- 二级缓存（earlySingletonObjects）存储早期暴露的半成品Bean，但无法处理需要生成代理对象的场景；
- 三级缓存（singletonFactories）存储`ObjectFactory`对象工厂，能在循环依赖中动态生成Bean的早期引用（包括代理对象），解决AOP代理场景下的循环依赖问题，因此功能更完整。

## 二、Spring循环依赖解决机制
### 1. 三级缓存定义
| 缓存名称 | 源码名称 | 作用 |
|----------|----------|------|
| 一级缓存 | `singletonObjects` | 单例池，缓存已完成完整生命周期、初始化完成的Bean对象 |
| 二级缓存 | `earlySingletonObjects` | 缓存早期暴露的半成品Bean对象（生命周期未走完） |
| 三级缓存 | `singletonFactories` | 缓存`ObjectFactory`对象工厂，用于创建Bean的早期引用 |

### 2. 解决流程（以A、B循环依赖为例）
1.  **实例化A**：调用构造函数创建A的原始对象，此时A处于半成品状态；
2.  **注册ObjectFactory**：将A的`ObjectFactory`存入三级缓存`singletonFactories`；
3.  **依赖注入A的属性B**：容器中不存在B，开始实例化B；
4.  **实例化B**：调用构造函数创建B的原始对象，此时B处于半成品状态；
5.  **注册ObjectFactory**：将B的`ObjectFactory`存入三级缓存`singletonFactories`；
6.  **依赖注入B的属性A**：从三级缓存中获取A的`ObjectFactory`，调用其`getObject()`方法生成A的早期引用（含代理对象），存入二级缓存`earlySingletonObjects`；
7.  **B初始化完成**：将B存入一级缓存`singletonObjects`；
8.  **A注入B**：将已初始化的B注入A；
9.  **A初始化完成**：将A存入一级缓存`singletonObjects`，并移除二、三级缓存中的A相关数据。

## 三、关键场景说明
### 1. 二级缓存能解决的场景
二级缓存可解决**普通对象（无AOP代理）**的循环依赖，通过提前暴露半成品对象，让依赖方获取引用完成注入。

### 2. 二三级缓存解决的场景
二三级缓存结合可解决**初始化过程中的循环依赖（第2-7步）**，特别是AOP代理场景：
- 三级缓存的`ObjectFactory`在调用`getObject()`时，会通过`BeanPostProcessor`生成目标对象的代理对象；
- 代理对象存入二级缓存，注入给依赖方，避免了直接暴露原始对象的问题。

### 3. 构造方法循环依赖的问题
构造方法注入的循环依赖，Spring无法自动解决：
- 构造函数是Bean生命周期的第一步，此时对象尚未实例化，无法存入缓存；
- 当A的构造函数依赖B，B的构造函数又依赖A时，会出现`Is there an unresolvable circular reference?`错误。

---

## 四、构造方法循环依赖的解决方案
使用`@Lazy`注解延迟加载依赖对象，让依赖对象在使用时才创建，打破构造函数的循环依赖：
```java
@Component
public class A {
    private B b;
    public A(@Lazy B b) {
        this.b = b;
    }
}

@Component
public class B {
    private A a;
    public B(@Lazy A a) {
        this.a = a;
    }
}
```

## 面试相关问题
1.  **问：什么是Spring循环依赖？**
    **答：** 循环依赖即两个或多个Bean互相持有对方，形成闭环依赖，如A依赖B，B依赖A。Spring通过三级缓存机制解决了大部分属性注入方式的循环依赖问题。

2.  **问：Spring三级缓存是如何解决循环依赖的？**
    **答：** 核心流程如下：
    1.  实例化A，创建原始对象并注册`ObjectFactory`到三级缓存；
    2.  注入A的依赖B，发现B未实例化，开始实例化B；
    3.  实例化B，创建原始对象并注册`ObjectFactory`到三级缓存；
    4.  注入B的依赖A，从三级缓存获取A的`ObjectFactory`，生成A的早期引用存入二级缓存；
    5.  B初始化完成，存入一级缓存；
    6.  将B注入A，A初始化完成，存入一级缓存，移除二、三级缓存中的A相关数据。

3.  **问：为什么构造方法注入的循环依赖Spring无法解决？**
    **答：** 构造函数是Bean生命周期的第一步，此时对象尚未实例化，无法存入缓存；当A的构造函数依赖B，B的构造函数又依赖A时，双方都无法完成实例化，导致死循环，因此Spring无法自动解决，需通过`@Lazy`注解延迟加载依赖。

4.  **问：三级缓存中为什么需要ObjectFactory？**
    **答：** 三级缓存中的`ObjectFactory`用于生成Bean的早期引用，特别是当Bean需要生成AOP代理对象时，`ObjectFactory`会调用`BeanPostProcessor`生成代理对象，确保注入给依赖方的是代理对象而非原始对象，解决了代理场景下的循环依赖问题。

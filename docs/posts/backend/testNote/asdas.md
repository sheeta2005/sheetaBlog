# Redis分片集群（Cluster集群）笔记
## 一、分片集群诞生背景
主从+哨兵只能实现**读高可用、故障自动切换**，无法解决两大痛点：
1. **海量数据存储**：所有数据都落在单个Master，单机内存上限受限
2. **高并发写入**：写操作全部打到单一主节点，写入性能瓶颈
   分片集群通过多主分片，分散数据与写入压力，同时搭配从节点保证高可用。

## 二、集群架构特征
1. **多Master分片**：集群包含多个主节点，**各Master分管不同数据**，实现数据分片存储、写请求分布式扩容。
2. **主从高可用**：每个Master可挂载若干Slave，Master宕机后，对应Slave自动晋升新主，依托内部心跳完成故障转移。
3. **节点互相探活**：全部Master之间通过`PING/PONG`心跳检测彼此在线状态。
4. **请求自动转发**：客户端可连接集群中任意节点，若当前节点不负责该Key槽位，节点返回`MOVED`指引客户端跳转至目标节点。

## 三、哈希槽（Slot）数据分片规则
### 1. 基础参数
Redis集群固定 **16384个哈希槽（0~16383）**，所有槽位均匀分配到集群各个Master节点。
### 2. Key路由计算公式
`CRC16(Key有效内容) % 16384 = 哈希槽编号`，根据槽号找到对应存储的Master。
### 3. Key有效部分（HashTag）规则
- 普通`key`：以**完整key字符串**作为CRC16计算内容
- `{aaa}name`：仅大括号内`aaa`作为有效内容，用于**哈希槽定向（相同大括号内容落到同一槽，实现key分组）**
> 用途：批量相关key落到同一个节点，可在集群下使用Redis事务、管道。

## 四、数据读写流程
1. 客户端发送读写指令，连接任意集群节点；
2. 节点提取key有效内容，通过`CRC16 % 16384`算出对应槽位；
3. 槽归属当前节点：直接执行命令，返回结果；
4. 槽归属其他节点：返回`MOVED 槽号 ip:port`，客户端重定向至目标节点访问。

## 五、面试高频问答
### 1. Redis分片集群的作用
1. 多Master分片存储，横向扩容存储容量与写入并发；
2. 每个Master挂载Slave，单主故障自动切换，保障高可用；
3. 节点间心跳监控集群健康，任意入口均可访问全量数据。

### 2. 分片集群数据存取原理
1. 集群预先划分16384个哈希槽，拆分分配给各个主节点；
2. 存取key时，截取key有效部分（大括号内容优先）；
3. CRC16哈希后对16384取模得到槽位，路由到槽所在Master完成读写。

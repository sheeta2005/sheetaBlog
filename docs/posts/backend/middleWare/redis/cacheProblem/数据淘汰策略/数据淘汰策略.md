# Redis内存淘汰策略（存入Redis/缓存基础.md）
## 一、定义
Redis配置`maxmemory`限定最大可用内存，**内存占满、新增key放不下时，按指定规则淘汰旧数据，这套规则就是内存淘汰策略**；和「过期删除（惰性+定期）」是两套独立机制：
- 过期删除：只清理**带TTL且到期**的key；
- 内存淘汰：内存爆满触发，可删全量key。

## 二、8种淘汰策略分类
### 1. noeviction（系统默认）
不淘汰任何key，内存打满后**所有新增写入命令直接报错**，读命令正常执行。

### 2. 限定只在带过期TTL的key里淘汰（volatile系列）
1. `volatile-ttl`：在设过期的key中，优先淘汰**剩余存活时间最短**的数据；
2. `volatile-random`：在带TTL的key里随机删；
3. `volatile-lru`：带TTL的key中，淘汰**最久没被访问**的数据；
4. `volatile-lfu`：带TTL的key中，淘汰**访问频次最低**的数据。

### 3. 全库所有key均可参与淘汰（allkeys系列）
1. `allkeys-random`：全部key随机淘汰；
2. `allkeys-lru`：全量key里淘汰最久未访问数据（**生产最常用**）；
3. `allkeys-lfu`：全量key里淘汰访问次数最少的数据。

### 算法名词释义
- **LRU(最近最少使用)**：对比最后一次访问时间，闲置越久越优先被删；
- **LFU(最少频次使用)**：统计key被访问次数，访问频次越低越优先淘汰。

## 三、落地选型建议
1. **常规业务（冷热数据分明，商品/资讯缓存）：allkeys-lru**
   保留近期热点数据，冷数据优先淘汰，项目标配。
2. **数据访问频率均匀、无明显冷热：allkeys-random**
   随机删除，算法开销最低。
3. **需要永久置顶部分热点（配置、字典）：volatile-lru**
   置顶数据不设置过期时间，不会进入淘汰池，仅清理带TTL的缓存。
4. **短时爆发热点（秒杀临时缓存、短期活动数据）：allkeys-lfu / volatile-lfu**
   短时高频访问的临时数据不会因为后期变冷立刻被淘汰。

## 四、LFU适用场景
1. **短期突发热点业务**：秒杀、限时活动、临时榜单缓存，部分key短期内被疯狂访问，活动结束后访问量骤降。LRU只看闲置时间，会把曾经的热点轻易清掉；LFU靠访问频次，能保留曾经高频、当前小幅降温的数据。
2. **周期性访问数据**：按月/周定时访问的统计缓存，平时少访问、固定时段高频，LFU靠历史访问计数保留，不会闲置几天就被LRU误删。
3. **长尾低频但有用的业务数据**：低频但不可丢失的配置类缓存，避免LRU因长时间闲置直接淘汰。

## 五、补充区分
过期策略管「到期自动删」，淘汰策略管「内存爆满被迫删」，二者互补。

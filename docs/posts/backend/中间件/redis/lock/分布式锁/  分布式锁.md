# Redis分布式锁完整笔记
## 一、落地使用场景
1. **库存扣减（秒杀/商品下单）**：防止超卖，分布式多实例并发抢占库存；
2. **定时任务防重复执行**：集群部署定时任务，避免多节点同时触发同一任务；
3. **接口幂等、重复请求拦截**：同一个订单重复提交，加锁保证单次处理；
4. **跨库事务串行化**：分布式多数据源更新，串行化避免数据错乱。

## 二、原生SET NX EX简易分布式锁原理
### 1.加锁
`SET lockKey 唯一标识 NX EX 过期时间`
- NX：key不存在才写入，实现互斥；
- EX：给key设置过期时间，防止死锁；
- value存入**当前线程唯一ID**，用于解锁校验，避免误删别人的锁。
### 2.解锁
**必须Lua脚本解锁，保证原子性**：先判断value和当前线程标识一致，再DEL删除；
> 不用Lua：判断和删锁分两步，中间锁过期会误删其他线程锁。

## 三、为什么必须设置过期时间
1. **防止死锁**：服务宕机、代码异常没执行解锁逻辑，过期时间到期Redis自动删锁，避免锁永久占用、其他线程永久拿不到锁；
2. 规避JVM卡死、网络异常导致解锁代码无法执行。

## 四、如何合理管控锁有效期（2种方案）
1. **预估业务耗时**：根据接口历史最大执行时长，过期时间 = 预估耗时×1.5倍，简单业务使用；
2. **看门狗续期（Redisson自动实现）**：
   加锁默认30s，后台看门狗定时（`releaseTime/3=10s`）执行续期Lua，业务没执行完自动刷新过期；**业务正常结束手动unlock，看门狗停止**。

## 五、Redisson分布式锁四大核心特性
### 1. 原子性：Lua脚本封装加/解锁
加锁、判断、续期、解锁全由单条Lua脚本执行，Redis单线程执行脚本天然原子，规避分步命令中间故障问题。
### 2. 可重入锁
底层采用**Hash数据结构**：
`key=锁名，field=线程唯一ID，value=重入计数`
- 同一线程重复加锁：计数+1；
- 重复解锁：计数-1，计数归零才真正删除锁；
  解决嵌套方法重复加锁死锁问题。
### 3. 自动看门狗续期
加锁成功启动后台定时任务，每1/3过期时间续期，业务结束销毁定时任务，不用手动管控过期时间。
### 4. 主从一致性：RedLock红锁（AP/CP选型）
#### （1）RedLock原理
多台独立Redis节点（无主从），客户端向**半数+1（N/2+1）**节点成功加锁才算获取锁成功；释放时逐个节点解锁，避免单节点宕机丢锁。
#### （2）CAP取舍
- **普通主从锁（主+从）：AP**：优先可用性，主宕机从晋升为主，但是主未同步锁数据到从，新线程可重新加锁，存在锁失效（牺牲一致性）；
- **RedLock红锁：CP**：优先一致性，必须过半节点加锁成功，少数节点故障不影响锁可靠性，节点多数故障则加锁失败（牺牲部分可用性）。

## 六、CAP&AP/CP思想补充
1. **CAP三原则**：分布式系统不能同时满足：
    - C一致性、A可用性、P分区容错；网络故障P必发生，只能二选一C/A。
2. **AP（可用性优先）：普通Redis主从锁**
   主宕机从自动上位可用，但锁数据没同步到从，出现锁丢失，放弃强一致、保可用。
3. **CP（一致性优先）：RedLock/Zookeeper临时节点锁**
   过半节点成功才算加锁，保证锁数据一致；集群过半故障时直接加锁失败，牺牲可用性。

## 七、原生SET锁缺陷&Redisson优化
|原生setnx锁短板|Redisson解决方案|
|----|----|
|手动写Lua保证原子，易错|内置Lua脚本|
|不可重入，嵌套加锁死锁|Hash结构计数实现可重入|
|过期时间难把控，业务超长提前过期|看门狗自动续期|
|主从切换锁丢失|RedLock红锁保证多节点一致性|

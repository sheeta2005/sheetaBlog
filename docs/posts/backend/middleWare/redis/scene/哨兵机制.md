# Redis Sentinel哨兵机制笔记
## 一、哨兵三大核心作用
1. **监控**：多个Sentinel节点通过心跳持续巡检Master、Slave实例运行状态，每秒发送PING探测。
2. **自动故障转移**：Master客观下线后，哨兵集群选举出新Slave晋升Master；原故障Master恢复上线后，自动降级为新Master的从节点。
3. **通知**：作为服务注册发现中心，主从切换后向业务客户端推送最新主节点地址，客户端自动切换连接。

## 二、下线判定规则（主观下线/客观下线）
### 1. 主观下线 SDOWN
单个Sentinel探测实例超时无PING响应，**仅当前哨兵标记该节点主观下线**，不代表真实故障。
### 2. 客观下线 ODOWN
满足**quorum法定票数**（哨兵半数以上）的哨兵均标记主节点主观下线，判定Master客观下线，触发故障转移。
> 配置建议：哨兵总数奇数（3/5个），quorum > 哨兵总数一半。

## 三、故障转移：Slave晋升Master四优先级规则（从上到下依次比对）
1. **断开时间**：与原Master失联超时的从节点直接淘汰，优先保留断线时间短的Slave。
2. **slave-priority（从库优先级）**：配置项`slaveof-priority`，数值**越小优先级越高**，默认100，0永远无法当选主节点。
3. **offset偏移量**：优先级相同时，`repl_backlog offset`数值越大，同步数据越完整，优先升级。
4. **实例运行ID**：前三项全部一致，实例运行ID字符串越小优先成为新主。

## 四、生产部署选型&面试问答
### 1. 常规架构选型
中小业务：**1主N从 + 3个哨兵**；单Redis实例内存控制10G以内，内存不足时按业务拆分多套独立主从+哨兵。

### 2. 脑裂问题（网络分区故障）
#### 脑裂成因
网络割裂：原Master和哨兵集群不在同一网段，哨兵探测不到旧Master，选举新Master；客户端仍向孤立旧Master写入数据，网络恢复后旧Master降级为从库，全量同步新Master数据，旧Master独有写入数据丢失。
#### 解决方案（配置参数）
`min-replicas-to-write` + `min-replicas-max-lag`：限制主节点必须至少N个从库同步延迟在指定毫秒内，不满足条件则主节点拒绝客户端写入，规避大量数据丢失。

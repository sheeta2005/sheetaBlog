# 布隆过滤器｜笔记
## 一、底层基础：Bitmap位图
**Bitmap = 二进制位数组**，数组里每一格只存`0 / 1`，1bit占1位空间，极度省内存。
- 0：代表该下标位置无数据；1：标记存在数据。

## 二、存入元素（以id=1举例）
1. 同一个数据，经过**3个不同Hash函数**，算出3个数组下标：`1、3、7`
2. 把位图下标`1、3、7`全部改成`1`
> 一条数据占用多个bit位，不是只占1格。

## 三、查询元素规则
1. **判定不存在：只要任意一个hash下标是0 → 数据一定不存在**（百分百准确）
2. **判定存在：全部下标都是1 → 大概率存在（存在误判）**
### 误判原因
不同key经过哈希，哈希下标撞在一起，全占了1；明明没存这个数据，但点位全1，误判存在。
> **特点：只可能误判存在，绝不会误判不存在**。

## 四、优缺点
### 优点
1. 占用内存极小，bit存储，海量数据也省空间
2. 查询速度极快，多次哈希+位运算
### 缺点
1. **无法删除数据**：一个bit位被多个元素共用，删1个会连带其他数据失效
2. 存在小概率误判（元素实际不存在，但判定为存在）

## 五、在Redis的用途：解决缓存穿透
1. 项目启动，把**数据库全部合法ID预先放进布隆过滤器**
2. 用户请求传id：
   - 过滤器判定不存在 → 直接拦截，不走Redis、不走MySQL
   - 过滤器判定存在 → 先查Redis，没命中再查DB



## 六、布隆过滤器两种落地实现（笔记存入：Redis/cacheProblem）
###一、Guava布隆过滤器（单机版）
#### 1.特点
1. **内存级、本地实现**：过滤器数据存JVM内存，不依赖Redis。
2. 局限：**不能跨服务共享**，集群部署时每个实例过滤器独立、数据不一致；项目重启数据全部丢失。
3. 适用：单体项目、小数据量。
#### 2.优点
开箱即用，Guava内置API，不用部署中间件。

### 二、Redisson布隆过滤器（分布式版，生产主流）
#### 1.特点
1. **基于Redis Bitmap实现**，过滤器数据存在Redis，**全集群所有服务共用同一个过滤器**。
2. 项目重启、多实例部署数据不丢失，完美适配分布式微服务。
#### 2.适用
分布式项目、防止缓存穿透（你业务场景首选）。

### 三、两者核心对比
|方案|存储位置|分布式支持|优缺点|
| ---- | ---- | ---- | ---- |
|Guava|JVM本地内存|❌不支持集群|实现简单，集群场景失效|
|Redisson|Redis位图|✅全服务共享|分布式标准方案，依赖Redis|

### 四、业务选型
1. **单体项目** → Guava
2. **微服务/分布式项目（商城等）** → Redisson

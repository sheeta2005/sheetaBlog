# 事务失效的场景

## 一、核心基础问题补充
### 1. 什么是检查异常/非检查异常？
- **检查异常（Checked Exception）**：继承自`Exception`但不继承`RuntimeException`的异常，编译器强制要求捕获或声明抛出（如`IOException`、`FileNotFoundException`）。
- **非检查异常（Unchecked Exception）**：继承自`RuntimeException`或`Error`的异常，编译器不强制处理（如`NullPointerException`、`ArithmeticException`）。

### 2. 为什么Spring事务默认只对RuntimeException生效？
Spring的声明式事务默认仅在抛出`RuntimeException`或`Error`时触发回滚，原因是：
- 检查异常通常是业务可预期的异常（如文件不存在），业务代码可以通过捕获处理，不一定需要回滚事务；
- 非检查异常通常是程序运行时的意外错误（如空指针、除零错误），这类异常通常无法通过业务代码恢复，需要事务回滚以保证数据一致性。

### 3. 为什么非public方法会导致事务失效？
Spring的声明式事务基于AOP实现，而AOP代理的创建前提是方法为`public`：
- JDK动态代理只能对接口的public方法生成代理；
- CGLIB代理也仅能对非final的public/protected方法进行增强；
- 非public方法无法被AOP代理拦截，因此事务通知无法生效，事务自然失效。

## 二、常见事务失效场景
### 场景1：异常被捕获未抛出
**代码示例**：
```java
@Transactional
public void update(Integer from, Integer to, Double money) {
    try {
        // 转账业务逻辑
        int a = 1 / 0; // 触发异常
    } catch (Exception e) {
        e.printStackTrace(); // 仅打印异常，未抛出
    }
}
```
**失效原因**：事务通知只有捕捉到目标方法抛出的异常，才能触发回滚；如果异常被业务代码捕获处理，事务通知无法感知到异常，因此不会回滚。
**解决方法**：在catch块中手动抛出`RuntimeException`，例如：
```java
catch (Exception e) {
    throw new RuntimeException(e);
}
```

---

### 场景2：抛出检查异常
**代码示例**：
```java
@Transactional
public void update(Integer from, Integer to, Double money) throws FileNotFoundException {
    // 转账业务逻辑
    new FileInputStream("test.txt"); // 抛出FileNotFoundException（检查异常）
}
```
**失效原因**：Spring事务默认仅对非检查异常（RuntimeException/Error）回滚，检查异常默认不触发回滚。
**解决方法**：通过`@Transactional(rollbackFor = Exception.class)`配置，指定所有异常都触发回滚。

---

### 场景3：非public方法添加@Transactional
**代码示例**：
```java
@Transactional(rollbackFor = Exception.class)
void update(Integer from, Integer to, Double money) { // 方法为包级私有
    // 转账业务逻辑
}
```
**失效原因**：Spring AOP代理仅对public方法生效，非public方法无法被代理拦截，事务通知无法执行。
**解决方法**：将方法改为`public`修饰。

## 面试相关问题
1.  **问：Spring中事务失效的场景有哪些？**
    **答：** 常见的事务失效场景包括：
    - 业务代码捕获了异常但未抛出，事务通知无法感知异常；
    - 抛出了检查异常（如IOException），而Spring默认仅对RuntimeException回滚；
    - `@Transactional`注解标注在非public方法上，AOP代理无法拦截该方法；
    - 事务方法被同类中其他方法直接调用，绕过了代理对象；
    - 数据库引擎不支持事务（如MyISAM）；
    - 事务传播行为配置不当，导致事务未按预期生效。

2.  **问：如何解决事务失效问题？**
    **答：** 对应场景的解决方法：
    - 异常被捕获时，在catch块中手动抛出RuntimeException；
    - 抛出检查异常时，配置`@Transactional(rollbackFor = Exception.class)`；
    - 非public方法改为public修饰；
    - 同类方法调用时，通过代理对象调用（如注入自身或使用AopContext）；
    - 确保数据库引擎为支持事务的InnoDB；
    - 合理配置事务传播行为，避免嵌套事务导致的失效。

3.  **问：事务失效的核心本质是什么？**
    **答：** Spring声明式事务基于AOP代理实现，事务生效的前提是方法被代理对象调用，且异常能被事务通知感知。所有事务失效场景，本质上都是破坏了这两个前提：要么代理未生效（如非public方法、同类调用），要么异常未被事务通知感知（如捕获异常、抛出检查异常）。
